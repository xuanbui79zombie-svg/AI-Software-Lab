# Development Workflow

本流程定义 Portfolio 项目从机会发现到开源发布的完整软件开发生命周期。每个阶段都必须产生可审查的证据，并满足退出条件后才能进入下一阶段。

## 基本原则

- 从真实用户问题和可验证价值出发，不以技术演示代替产品目标。
- 优先交付范围清晰的最小可行版本，再根据证据迭代。
- 将测试、安全、文档、部署和可维护性视为开发的一部分。
- 使用 `TASKS.md` 管理 TODO、DOING、DONE 和 BLOCKED，并采用 P0–P3 优先级。
- 重要决策、验收结果和风险必须可追踪，不依赖口头记忆。
- 任何阶段发现关键假设不成立时，允许返回前一阶段重新决策。

## 阶段总览

| 阶段 | 核心问题 | 主要负责人 | 必需产物 |
| --- | --- | --- | --- |
| 1. 项目发现 | 是否值得做？ | Portfolio Manager | 机会说明、价值评分、初步范围 |
| 2. 需求分析 | 用户到底需要什么？ | Product Manager | 用户、问题、需求、成功指标 |
| 3. 产品设计 | 用户如何完成目标？ | Product Manager | 用户流程、MVP、验收标准 |
| 4. 架构设计 | 如何可靠地实现？ | Software Architect | 架构、API、数据、技术决策 |
| 5. 开发 | 如何逐步交付？ | Engineering | 可运行代码、测试、文档 |
| 6. 测试 | 是否达到发布质量？ | QA Engineer | 测试报告、缺陷、发布结论 |
| 7. 部署 | 如何安全上线和恢复？ | Engineering | 部署、监控、回滚、运行证据 |
| 8. 开源包装 | 如何让外部用户理解和参与？ | Maintainer | README、许可证、发布与贡献指南 |

## 阶段 1：项目发现

### 目标

从职业目标、用户问题和市场机会中筛选值得投入的项目，避免创建重复、无法完成或缺少展示价值的仓库。

### 关键活动

- 明确目标岗位、目标用户、问题场景和现有替代方案。
- 收集问题证据，区分事实、假设和未知项。
- 评估技术价值、商业价值、简历价值、实现成本和风险。
- 定义粗略 MVP、可演示结果和停止条件。
- 在 `PROJECT_ROADMAP.md` 中记录候选项目和 P0–P3 优先级。

### 必需产物

- 一句话机会说明。
- 目标用户和问题证据。
- 技术、商业和简历价值评分。
- 时间、资源、依赖和主要风险。
- 启动、验证、暂停或拒绝建议。

### 退出条件

- [ ] 问题和目标用户清晰。
- [ ] 项目与 Portfolio 使命和目标岗位相关。
- [ ] MVP 可以在可接受周期内完成。
- [ ] 有可验证成功指标和 Demo 计划。
- [ ] 已决定是否进入需求分析。

## 阶段 2：需求分析

### 目标

把机会转化为可验证、可排序、可验收的产品需求。

### 关键活动

- 定义用户角色、使用场景、痛点和结果。
- 绘制当前流程和期望流程。
- 编写功能需求、非功能需求和边界条件。
- 识别认证、权限、数据、隐私、合规和可访问性要求。
- 为需求分配 ID、优先级和验收标准。

### 必需产物

- `docs/PRODUCT.md` 初稿。
- 用户流程与关键用例。
- P0–P3 需求表。
- 成功指标、验收标准和非目标。
- 风险、假设和开放问题。

### 退出条件

- [ ] 每个 P0/P1 需求都有验收标准。
- [ ] 非目标和范围边界明确。
- [ ] 数据、安全和质量需求已识别。
- [ ] 没有会阻止产品设计的关键未知项。

## 阶段 3：产品设计

### 目标

定义用户如何完成核心任务，并将需求收敛为一致、可实现的 MVP 体验。

### 关键活动

- 设计核心用户旅程、信息结构和交互状态。
- 覆盖空状态、加载、错误、无权限和恢复场景。
- 定义内容、反馈、可访问性和响应式要求。
- 使用原型或流程图验证关键假设。
- 根据用户价值和实现成本调整 MVP。

### 必需产物

- 核心用户流程或原型。
- 页面、状态和交互说明。
- MVP 范围、非目标和发布后候选项。
- 更新后的验收标准和可用性指标。

### 退出条件

- [ ] 核心流程从入口到结果完整。
- [ ] 错误和边界状态已定义。
- [ ] 产品、设计和工程对 MVP 范围理解一致。
- [ ] 产品设计足以支持架构决策。

## 阶段 4：架构设计

### 目标

选择满足当前需求的最简单可靠架构，并明确系统边界、契约、质量属性和演进方式。

### 关键活动

- 绘制系统上下文、组件、数据流和信任边界。
- 定义 API、数据库、外部服务和 AI 能力契约。
- 评估性能、可靠性、安全、成本、扩展性和可观测性。
- 选择技术栈并记录替代方案和权衡。
- 规划部署、迁移、降级、回滚和恢复。
- 完成初步威胁建模并检查 `SECURITY_CHECKLIST.md`。

### 必需产物

- `docs/ARCHITECTURE.md`。
- `docs/API.md` 和 `docs/DATABASE.md`，不适用时明确标记。
- `TECH_STACK.md`。
- ADR、威胁模型和验证任务。

### 退出条件

- [ ] 组件职责和依赖方向明确。
- [ ] 关键接口、数据和失败行为明确。
- [ ] 架构满足已确认的质量目标。
- [ ] 高风险假设有验证计划。
- [ ] 开发任务可以按垂直切片拆分。

## 阶段 5：开发

### 目标

通过小批量、可测试、可评审的变更持续交付端到端功能。

### 关键活动

- 将需求拆成带验收标准的任务并维护 `TASKS.md`。
- 使用短生命周期分支和单一目的提交。
- 按垂直切片实现 UI、服务、数据和集成。
- 同步编写自动化测试、日志、错误处理和文档。
- 运行格式化、静态分析、单元和集成检查。
- 使用 `CODE_REVIEW_CHECKLIST.md` 进行自查和评审。

### 必需产物

- 可运行的源代码和配置示例。
- 自动化测试和检查结果。
- 更新后的 README、API、架构、任务和 Changelog。
- 可审查 Pull Request 和风险说明。

### 退出条件

- [ ] P0/P1 范围已实现并满足验收标准。
- [ ] 核心自动化测试通过。
- [ ] 没有已知 P0/P1 缺陷。
- [ ] 代码评审和安全自查完成。
- [ ] 功能已准备进入独立测试。

## 阶段 6：测试

### 目标

基于风险独立验证产品行为和质量属性，为发布提供证据。

### 关键活动

- 建立需求到测试的追踪矩阵。
- 执行成功、边界、失败、权限、恢复和回归场景。
- 验证性能、安全、可访问性和兼容性目标。
- 记录缺陷、复现步骤、严重度和修复证据。
- 复测修复并评估剩余风险。

### 必需产物

- 测试计划、场景矩阵和执行结果。
- 缺陷列表及修复状态。
- 自动化覆盖和未测试范围。
- 发布建议：通过、有条件通过或阻断。

### 退出条件

- [ ] 所有 P0/P1 测试场景通过。
- [ ] P0/P1 缺陷已解决或发布被阻断。
- [ ] 剩余风险已记录并被接受。
- [ ] 发布候选版本可重复构建。

## 阶段 7：部署

### 目标

以可重复、可观测、可回滚的方式发布，并验证真实环境中的核心功能。

### 关键活动

- 固定版本、依赖和构建产物。
- 在预发布环境验证配置、迁移和部署步骤。
- 备份关键数据并确认回滚条件。
- 使用最小权限管理环境、凭据和服务账号。
- 部署后执行健康检查、冒烟测试和关键指标验证。
- 配置日志、指标、告警、成本和错误监控。

### 必需产物

- 部署与回滚说明。
- 环境变量和配置文档，不包含真实秘密。
- 迁移、备份和恢复记录。
- 发布版本、运行证据和监控链接。

### 退出条件

- [ ] 部署可重复且核心冒烟测试通过。
- [ ] 监控、告警和日志可用。
- [ ] 回滚或降级路径已验证。
- [ ] Demo 可以访问或已有可信演示材料。

## 阶段 8：开源包装

### 目标

让外部用户能够理解、运行、评估和贡献项目，并将成果转化为 Portfolio 证据。

### 关键活动

- 完善项目说明、功能、架构、安装、运行、测试和 Demo。
- 添加许可证、贡献指南、行为准则和安全报告方式。
- 配置 Issue/PR 模板、标签、里程碑和发布说明。
- 清除秘密、私人数据、调试文件和不必要生成物。
- 验证全新环境安装和文档链接。
- 更新 `PORTFOLIO_STATUS.md`、简历价值评分和项目证据。

### 必需产物

- 高质量 README 和完整文档导航。
- LICENSE、CONTRIBUTING、CODE_OF_CONDUCT 和 SECURITY 文档（按项目需要）。
- Changelog、语义版本和发布说明。
- Demo、截图或视频，以及可验证结果。
- 贡献入口和维护计划。

### 退出条件

- [ ] 新用户可仅依赖文档运行项目。
- [ ] 仓库无秘密、敏感数据和无效链接。
- [ ] 许可证、贡献和安全政策清晰。
- [ ] 发布版本和 Demo 可验证。
- [ ] 项目已在 Portfolio 状态表中更新。

## 变更与回退规则

- 产品价值不成立时，返回阶段 1 或停止项目。
- 需求或范围发生重大变化时，返回阶段 2–3 并重新评估架构。
- 架构假设失败时，更新 ADR 并返回阶段 4。
- 测试或部署发现阻断风险时，返回阶段 5 修复，不绕过门禁。
- 所有回退都必须更新任务、文档、风险和验收条件。
