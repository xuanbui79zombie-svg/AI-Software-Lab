# Portfolio Mission

## 长期目标

成为具备商业项目开发能力的独立开发者：能够从真实问题出发，独立完成需求分析、方案设计、开发测试、部署交付、数据验证与持续迭代，并用可验证的成果获得工作机会、客户信任和开源影响力。

## 1. Portfolio 目标

- 建立以成果和证据为核心的个人开发者 Portfolio，而不是简单堆积代码仓库。
- 展示从问题发现到产品交付的完整能力，包括技术实现、用户体验、部署和迭代。
- 为简历、求职面试、自由职业和开源协作提供可信的项目材料。
- 逐步形成稳定的工程方法、技术方向和个人开发者品牌。
- 保持少而精的项目组合，优先展示 3–5 个高质量代表项目。

## 2. 项目选择标准

候选项目应尽量满足以下条件：

- **真实问题**：目标用户、使用场景和痛点清晰，不以技术演示代替需求。
- **可验证价值**：能够通过效率、准确率、成本、用户反馈或使用数据验证结果。
- **可演示交付**：可以提供可访问 Demo、截图、视频或清晰的本地运行方式。
- **技术成长**：至少强化一项可迁移能力，例如系统设计、AI 集成、数据处理、测试或部署。
- **范围可控**：最小可行版本边界清楚，能够在合理周期内完成并维护。
- **简历相关**：能够支持目标岗位所需要的能力证明。
- **可公开复用**：不包含敏感数据，适合开源，或能沉淀为可复用的技术经验。

不满足真实需求、无法完成、无法演示或与目标岗位无关的项目，不进入开发阶段。

## 3. 技术成长路线

1. **工程基础**：Git、命令行、数据结构、网络基础、调试、代码规范与技术文档。
2. **应用开发**：前端、后端、数据库、API、身份认证和完整业务流程。
3. **质量工程**：自动化测试、日志、错误处理、安全、性能和持续集成。
4. **AI 软件能力**：模型 API、提示设计、检索增强、评估、成本控制和可靠性治理。
5. **部署与运维**：容器化、云部署、监控、版本发布和故障恢复。
6. **商业交付**：需求澄清、范围管理、用户反馈、数据指标、迭代和维护。
7. **开源与影响力**：规范协作、Issue、Pull Request、技术写作和社区贡献。

## 4. 项目质量要求

每个进入 Portfolio 的项目至少应具备：

- 清晰的问题定义、目标用户、范围和成功指标。
- 完整 README，包括功能、架构、技术栈、安装、配置、运行和使用说明。
- 可复现的开发环境，以及不泄露密钥的配置方式。
- 清晰的代码结构、统一的格式和必要的类型或接口约束。
- 覆盖核心流程的自动化测试和明确的验证结果。
- 合理的错误处理、日志、安全和隐私保护措施。
- 可访问 Demo，或可信的演示视频、截图和操作说明。
- 明确的许可证、版本记录、已知限制和后续改进方向。
- 能够解释关键技术决策、权衡、失败经验和最终结果。

## 5. 简历展示标准

项目进入简历前必须满足以下要求：

- 用一句话说明服务对象、解决的问题和最终价值。
- 明确个人职责，避免只罗列团队或工具名称。
- 使用数据、性能、准确率、用户反馈或交付结果说明影响。
- 列出真正使用且能够解释的核心技术栈。
- 同时提供 GitHub 地址和 Demo 或演示材料。
- 能够在面试中讲清背景、任务、行动、结果和技术权衡。
- 代码、文档、提交记录和演示内容保持一致且可验证。
- 只有完成度高、差异化明显、与目标岗位相关的项目才作为重点项目。
