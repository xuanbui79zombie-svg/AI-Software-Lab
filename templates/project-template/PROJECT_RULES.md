# Project Rules

## 架构规则

- 业务逻辑、接口适配、数据访问和外部服务集成必须保持清晰边界。
- 依赖方向应由外层指向稳定的核心层，核心逻辑不直接依赖框架细节。
- 模块应围绕单一职责组织，公开接口最小化，避免隐式跨层访问。
- 外部 API、数据库、文件系统和模型服务必须通过可替换的适配层接入。
- 关键架构决策记录在 `docs/ARCHITECTURE.md`，数据契约和 API 契约分别记录在对应文档。
- 优先采用满足当前需求的最简单架构，不为未确认的未来需求过度设计。

## 代码规范

- 使用项目选定语言和框架的官方格式化与静态检查工具。
- 命名必须表达业务含义，避免无意义缩写和模糊的通用名称。
- 函数和模块保持小而聚焦；复杂逻辑拆分为可测试单元。
- 公共接口、复杂算法和非显而易见的权衡必须有必要说明。
- 错误必须显式处理并提供可诊断信息，不静默吞掉异常。
- 配置通过环境或配置文件注入，不在代码中硬编码环境差异和秘密。
- 新增依赖前说明用途、维护状态、安全风险和替代方案。

## 修改流程

1. 明确任务、优先级、验收标准和非目标。
2. 检查关联产品、架构、数据和 API 文档。
3. 识别受影响模块、兼容性、安全和迁移风险。
4. 先设计最小修改，再实现代码和测试。
5. 运行适当检查并记录结果。
6. 更新文档、任务状态和变更日志。
7. 自查差异后提交评审。

## 禁止事项

- 禁止提交密钥、令牌、密码、生产数据或个人敏感信息。
- 禁止绕过认证、授权、输入校验或安全控制来加快开发。
- 禁止在没有迁移、备份和回滚方案时破坏性修改数据结构。
- 禁止修改已发布 API 契约而不说明兼容性影响。
- 禁止通过跳过测试、删除断言或忽略错误来制造“通过”结果。
- 禁止引入未使用的依赖、复制实现或无任务依据的大规模重构。
- 禁止直接修改生成文件或第三方代码，除非项目文档明确允许。
- 禁止在未检查工作区和差异的情况下批量暂存或提交文件。
