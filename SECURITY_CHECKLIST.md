# Security Checklist

用于设计、代码评审和发布前的安全自查。根据项目风险选择适用项；高风险系统仍需要专业威胁建模、渗透测试和合规评审。

## 1. 安全范围与威胁模型

- [ ] 识别资产、用户、数据、外部系统和信任边界。
- [ ] 列出攻击者能力、滥用场景和最坏影响。
- [ ] 对认证、支付、个人数据、文件上传、AI 工具调用等高风险流程单独建模。
- [ ] 区分必须修复、可接受和需持续监控的风险。
- [ ] 为关键控制定义可验证要求，而不是只写“保持安全”。

## 2. 仓库与访问控制

- [ ] 仅向需要的人员和自动化授予最小权限。
- [ ] 默认分支使用 ruleset 或分支保护，并要求评审与状态检查。
- [ ] 管理员绕过、强制推送、删除分支和发布权限受到限制。
- [ ] 启用适用的依赖图、Dependabot、代码扫描和秘密扫描功能。
- [ ] 提供 `SECURITY.md` 和私密漏洞报告方式。
- [ ] 定期审查协作者、Deploy Key、OAuth App、GitHub App 和 Actions 权限。

## 3. 密钥与配置

- [ ] 代码、历史、日志、测试数据和文档中没有真实密钥或令牌。
- [ ] 使用秘密管理系统或 GitHub Actions Secrets，不在配置文件硬编码。
- [ ] 开发、测试和生产使用不同凭据与权限。
- [ ] 凭据具有最小权限、有效期和轮换策略。
- [ ] 暴露凭据立即撤销并轮换，不能只从最新提交删除。
- [ ] 提供不含秘密的 `.env.example` 或配置说明。

## 4. 认证与会话

- [ ] 使用成熟的身份方案和安全密码存储，不自制密码算法。
- [ ] 高风险账号支持或要求 MFA。
- [ ] 登录、重置、注册和账号恢复防止枚举与滥用。
- [ ] 会话 ID 随机、不可预测，并在登录、提权和退出时正确轮换或失效。
- [ ] Cookie 使用适当的 `Secure`、`HttpOnly` 和 `SameSite` 属性。
- [ ] 访问令牌有合理有效期、撤销和刷新策略。

## 5. 授权与权限

- [ ] 每个受保护操作在服务端验证身份、资源和动作权限。
- [ ] 默认拒绝，不能依赖 UI 隐藏、路由名称或客户端角色判断。
- [ ] 对象 ID 访问防止 IDOR / BOLA，验证资源归属或授权关系。
- [ ] 批量、导出、管理和跨租户操作具有更严格权限。
- [ ] 服务账号、API Key 和自动化令牌使用最小范围。
- [ ] 权限变化和高风险操作留有审计记录。

## 6. 输入、输出与命令执行

- [ ] 在可信边界验证所有外部输入的类型、长度、格式、范围和数量。
- [ ] 数据库查询、系统命令和模板使用安全参数化接口。
- [ ] HTML、URL、JavaScript、日志和其他输出按上下文编码。
- [ ] 文件上传限制类型、大小、名称、存储位置和访问方式，并进行内容检查。
- [ ] 防止路径遍历、SSRF、不安全反序列化和任意代码执行。
- [ ] 错误响应不暴露堆栈、内部路径、查询、密钥或实现细节。

## 7. Web 与 API

- [ ] 全程使用 TLS，禁用不安全协议与弱配置。
- [ ] 状态改变请求具有适当 CSRF 防护。
- [ ] CORS、重定向、嵌入和内容安全策略仅允许必要来源。
- [ ] API 有认证、授权、限流、配额、超时和请求大小限制。
- [ ] 敏感操作支持幂等、防重放或二次确认。
- [ ] OpenAPI / Schema 与实际验证规则保持一致。

## 8. 数据保护与隐私

- [ ] 只收集和保留实现目标所需的数据。
- [ ] 对数据进行分类，明确敏感字段、所有者和生命周期。
- [ ] 敏感数据在传输和存储中使用适当加密。
- [ ] 密钥与加密数据分离管理，并有轮换和恢复方式。
- [ ] 日志、分析、备份和错误报告进行脱敏。
- [ ] 删除、导出、备份、恢复和保留流程符合产品及合规要求。

## 9. 数据库

- [ ] 使用独立、最小权限数据库账号，不让应用拥有不必要管理权限。
- [ ] Schema 约束保护关键不变量，不只依赖应用层检查。
- [ ] 迁移包含备份、回滚、兼容窗口和数据验证。
- [ ] 防止 SQL/NoSQL 注入、越权查询和大范围意外更新。
- [ ] 备份加密、访问受控，并定期验证恢复。

## 10. 依赖与供应链

- [ ] 依赖来自可信来源，并固定可复现版本或锁文件。
- [ ] 新依赖评估维护状态、许可证、权限、体积和替代方案。
- [ ] 使用依赖审查和漏洞告警，并为高风险漏洞定义响应时限。
- [ ] 构建、发布和安装脚本不执行来源不明代码。
- [ ] GitHub Actions 固定可信 action 版本或提交 SHA，并限制令牌权限。
- [ ] 发布产物可追踪到源码、版本、构建和审批记录。

## 11. AI 功能（适用时）

- [ ] 将用户内容、网页、文档、检索结果和模型输出视为不可信输入。
- [ ] 防护直接和间接提示注入，不让内容覆盖系统权限规则。
- [ ] 模型工具调用经过结构验证、权限检查、范围限制和操作确认。
- [ ] 不把秘密、个人数据或未授权内容发送给模型提供商。
- [ ] 限制模型代理权限、步骤、成本、资源和可执行动作。
- [ ] 对高风险输出使用确定性规则、引用、人工复核或拒绝策略。
- [ ] 评估敏感信息泄露、供应链、过度依赖和模型不可用风险。

## 12. 日志、监控与响应

- [ ] 记录认证失败、权限拒绝、配置变化和高风险操作。
- [ ] 日志包含时间、请求或追踪 ID，但不包含秘密和敏感正文。
- [ ] 为异常流量、错误率、资源耗尽和安全事件设置告警。
- [ ] 时间同步、日志保留和访问权限满足调查需要。
- [ ] 定义漏洞报告、分级、遏制、修复、通知和复盘流程。
- [ ] 安全事件后轮换凭据、检查影响范围并补充回归控制。

## 13. 部署与基础设施

- [ ] 开发、测试、预发布和生产环境隔离。
- [ ] 运行时使用非特权身份、最小网络访问和只需文件权限。
- [ ] 容器、系统和运行时使用受支持版本并及时修补。
- [ ] 基础设施和策略变更经过评审、计划和回滚验证。
- [ ] 管理接口、调试模式和测试端点不暴露到生产。
- [ ] 资源限制、防滥用和成本告警已配置。

## 14. 发布安全门禁

- [ ] P0/P1 安全问题已关闭或发布被阻断。
- [ ] 代码、依赖、秘密和配置扫描结果已审查。
- [ ] 高风险流程测试、代码评审和威胁模型已完成。
- [ ] 备份、回滚、监控和事件联系人可用。
- [ ] 已知风险、缓解措施、责任人和复评日期已记录。
- [ ] 对外安全说明和漏洞报告渠道已准备。
