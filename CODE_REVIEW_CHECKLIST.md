# Code Review Checklist

用于 Pull Request 自查和独立评审。只勾选已检查并有证据支持的项目；不适用项标记 `N/A` 并说明原因。

## 评审信息

- PR / Commit：`<LINK_OR_SHA>`
- 变更目标：`<CHANGE_GOAL>`
- 关联任务：`<TASK_ID>`
- 风险等级：`Low | Medium | High | Critical`
- 评审者：`<REVIEWER>`
- 检查日期：`<YYYY-MM-DD>`

## 1. 范围与需求

- [ ] 变更与任务、需求和验收标准一致。
- [ ] 没有夹带无关重构、格式化或依赖升级。
- [ ] 非目标和不支持场景已明确。
- [ ] 用户可见变化、兼容性和迁移影响已说明。
- [ ] 高风险假设和开放问题已记录。

## 2. 正确性

- [ ] 核心成功路径正确实现。
- [ ] 空值、边界、重复、无效输入和错误路径已处理。
- [ ] 状态转换、并发、重试和幂等行为正确。
- [ ] 时间、时区、精度、排序和分页语义明确。
- [ ] 错误不会被静默吞掉，诊断信息足够且不泄密。
- [ ] 回滚、降级或失败恢复行为可接受。

## 3. 架构与可维护性

- [ ] 组件职责、依赖方向和公开接口符合架构。
- [ ] 业务逻辑没有泄漏到 UI、路由或基础设施层。
- [ ] 复用了现有模式，没有重复实现相同能力。
- [ ] 复杂度与当前需求匹配，没有明显过度设计。
- [ ] 命名表达业务意图，函数和模块职责聚焦。
- [ ] 关键技术权衡已记录，文档与实现一致。

## 4. 安全与隐私

- [ ] 已完成适用的 `SECURITY_CHECKLIST.md` 项目。
- [ ] 认证和授权在服务端或可信边界内强制执行。
- [ ] 所有外部输入经过类型、长度、格式和权限验证。
- [ ] 输出已正确编码，查询和命令使用安全参数化方式。
- [ ] 没有提交密钥、令牌、密码、个人数据或生产配置。
- [ ] 日志、错误和分析数据不会暴露敏感信息。
- [ ] 新依赖、工作流和外部服务经过来源与权限审查。

## 5. API 与兼容性

- [ ] API 请求、响应、状态码和错误格式符合契约。
- [ ] 破坏性变更有版本、迁移或兼容窗口。
- [ ] 分页、过滤、排序、限流、超时和重试行为明确。
- [ ] 客户端与服务端类型和验证规则一致。
- [ ] 外部 API 失败、降级和配额耗尽已处理。

## 6. 数据库与数据安全

- [ ] Schema、约束、索引和数据生命周期符合设计。
- [ ] 迁移可重复、可回滚，并评估锁表与数据量风险。
- [ ] 写操作具有正确事务边界和一致性保证。
- [ ] 查询计划、N+1、全表扫描和大结果集风险已评估。
- [ ] 数据回填、删除和保留策略安全且可审计。
- [ ] 备份和恢复要求未被破坏。

## 7. 前端与用户体验

- [ ] 加载、空状态、错误、无权限和成功反馈完整。
- [ ] 表单验证、提交状态和重复操作行为正确。
- [ ] 键盘操作、焦点、语义标签和颜色对比可接受。
- [ ] 响应式布局和目标浏览器行为已验证。
- [ ] 用户输入不会造成 XSS、开放重定向或不安全渲染。
- [ ] 性能敏感的资源、渲染和网络请求合理。

## 8. AI 功能（适用时）

- [ ] 模型、提示、检索和工具输入输出契约明确。
- [ ] 有可重复评估，而非仅依赖主观示例。
- [ ] 提示注入、敏感信息、越权工具调用和过度代理已控制。
- [ ] 超时、重试、成本、限额、降级和人工复核机制完整。
- [ ] 模型与提示版本可追踪，失败输出不会直接驱动高风险操作。

## 9. 测试

- [ ] 新功能覆盖核心成功路径。
- [ ] Bug 修复包含能够复现问题的回归测试。
- [ ] 权限、边界、失败和恢复路径有适当覆盖。
- [ ] 测试稳定、隔离且不依赖未声明环境状态。
- [ ] 没有被无理由删除、跳过或弱化的测试。
- [ ] 实际运行的检查、命令和结果已记录。

## 10. 性能与可靠性

- [ ] 热路径的时间、内存、网络和数据库成本可接受。
- [ ] 缓存具有正确失效和一致性策略。
- [ ] 超时、重试和退避不会造成重试风暴或重复副作用。
- [ ] 资源、连接、文件和锁能够正确释放。
- [ ] 日志、指标和告警足以诊断关键失败。

## 11. 文档与运营

- [ ] README、产品、架构、API、数据库或技术栈文档已按影响更新。
- [ ] 配置、迁移、部署和回滚步骤可执行。
- [ ] `CHANGELOG.md` 记录用户或维护者可见变化。
- [ ] `TASKS.md` 状态与实际进度一致。
- [ ] 已知限制和后续事项已记录。

## 12. Git 与 Pull Request

- [ ] 提交信息清晰，每个提交只有一个逻辑目的。
- [ ] PR 描述说明变更、原因、影响、验证和风险。
- [ ] 差异中没有生成物、依赖目录、临时文件或调试代码。
- [ ] 分支基于正确目标，且没有意外覆盖其他人的修改。
- [ ] 所有必需检查通过，阻断评论已解决。

## 评审结论

- [ ] **Approve**：没有阻断问题，风险和证据可接受。
- [ ] **Request changes**：存在必须修复的问题。
- [ ] **Block**：存在 P0/P1 安全、数据或正确性风险。

### 发现

按严重度记录：`[P0–P3] 标题 — 文件:行 — 影响 — 建议修复 — 必要测试`

没有发现时，明确写明“未发现阻断问题”，并列出未验证范围和剩余风险。
